Development Process Guide

The life of a release: DEV → TST → ACC → PRD

Release Lifecycle

Every release follows a structured path through four environments. Each environment has its own purpose, statuses, and actions. This guide explains what happens at every stage so any developer can understand exactly where a release is and what needs to happen next.

Environments

Local development environment. Developers work on feature branches and push code here freely. DEV is loosely managed — no formal status tracking or release gates apply.

Test environment. The develop branch holds all work for the next ACC release. Developers create, merge & review PRs to develop during this period.

Acceptance environment. A release branch is cut from develop, reviewed, and deployed. Multiple ACC rounds can happen for the same release. Only the last round advances to PRD.

Production. The final release branch is merged into main and develop after a successful deploy.

TST

Test — Development Phase

Development

The develop branch holds all development for the next ACC release. During this period developers should be creating, merging and reviewing PRs to develop.

What to do:

  • Create feature branches from develop
  • Open PRs targeting develop
  • Review and merge PRs before the deadline

Deadline: 12:00 on the first ACC release day. All PRs must be merged to develop before this time.

Duration: From the first TST date (e.g. 26/03) until 12:00 on the first ACC release day.

ACC

Acceptance — Release to ACC

Setup ACC

At 12:00 on the ACC release day, the ACC setup begins.

Steps:

  1. 1 Revert all commits on Jira that are not part of this ACC release on develop
  2. 2 Branch off from develop → release/xx.xx.x (e.g. release/26.04.0)
  3. 3 Create a PR from release/xx.xx.x → develop

ACC Review

From 12:15 to 16:00, the release PR must be reviewed and approved.

Since features were already reviewed on develop during TST, this review should be quick — focus on the release scope and any revert correctness.

To Release in ACC

After PR approval, the deployment happens at the scheduled time on the ACC release day (e.g. 17:00). The exact time can vary per release.

ACC Done

After a successful deploy, the PR is approved and merged.

ACC can repeat multiple times

There can be multiple ACC release moments for the same release version. Each round goes through Setup ACC → ACC Review → To Release in ACC → ACC Done. Only the last ACC round is used to deploy to PRD and merge into main.

PRD

Production — Release to PRD

Setup PRD

On the Friday before the PRD release week, the production setup begins.

Steps:

  1. 1 Create PR from release/xx.xx.x → develop
  2. 2 Create PR from release/xx.xx.x → main
  3. 3 Create a Jira ticket with the infra team for the PRD deployment

PRD Review

The PRs must be reviewed and approved during the period from the Friday before PRD until before the PRD deploy moment.

To Release in PRD

The deploy happens at the agreed-upon moment. The date and time are variable per release.

Completed

After a successful deploy, the PRs are merged into both main and develop. The release is done.

Note: Only the last ACC release branch is used for the PRD deployment and merge into main.

Quick Reference

Key Branches

main Production code — always matches what's deployed in PRD
develop Integration branch — holds development for the next ACC release
release/xx.xx.x Release branch — cut from develop at ACC setup time
feature/* Feature branches — created by developers, merged to develop

Key Times on ACC Day

12:00 Development deadline — ACC setup starts
12:15 Review period opens
16:00 Review period closes
17:00* ACC deploy (time varies per release)
